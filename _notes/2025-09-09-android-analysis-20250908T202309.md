## Task 3 | Android Architecture - An Overview

Before diving into the details of Android analysis, it's important to familiarize yourself with the Android architecture and how its components are layered.

 Android Architecture Android's architecture is layered and modular, making it versatile and complex from a forensics standpoint.

![Image 3](https://tryhackme-images.s3.amazonaws.com/user-uploads/5e8dd9a4a45e18443162feab/room-content/5e8dd9a4a45e18443162feab-1747826237632.svg)

 

 It consists of the following layers:

 **Linux Kernel Layer**

 The Linux kernel is the foundation of Android's architecture, providing core operating system functionalities. It acts as the abstraction layer between the hardware and the software stack.

 
- - Core system services
- Process management
- Memory management
- Device drivers

 **Native Libraries**

 The native libraries provide core functionality to the Android system and applications.

 
- - - System libraries
- Media frameworks
- SQLite database engine
- WebKit browser engine

 **Application Framework Layer**

 The application framework layer provides APIs for higher-level services like location, telephony, and resource management. It enables developers to manage apps and components like Activities, Services, and Content Providers.

 
- - APIs for application development
- Resource management
- Component interaction
- Service handling

 **Application Layer**

 This is the topmost layer in the Android architecture. It includes system and user-installed apps, the primary source of user data in a forensic investigation.

 
- - User applications
- System applications
- Third-party apps

 Android Filesystem Android uses a Linux-based filesystem that is layered and partitioned to manage data securely and efficiently. The structure is customized based on the device's hardware and firmware. Due to the filesystem structure, the filesystem of every device plays a crucial role in digital forensics, directly influencing how data is stored, accessed, and recovered.

 ![Image 4](https://tryhackme-images.s3.amazonaws.com/user-uploads/5e8dd9a4a45e18443162feab/room-content/5e8dd9a4a45e18443162feab-1747796686545.png)

 

 FileSystem Partitions In the Android OS, the filesystem is in the following structure:

   Android Filesystem Structure 
```Android Filesystem Structure 
├── system/                  → Android OS system files (read-only in user mode)
│   ├── bin/                 → System binaries
│   ├── lib/                 → Shared libraries
│   └── framework/           → Java framework .jar files
│
├── data/                    → Main user data partition
│   ├── app/                 → Installed APKs
│   ├── data/                → App private data
│   ├── misc/                → Misc system info (e.g., WiFi configs)
│   ├── media/               → Encrypted storage mount point
│   └── system/              → User accounts, settings
│
├── sdcard/ (or /storage/emulated/0) → User files, photos, downloads
│
├── vendor/                 → OEM-specific binaries/libraries
│
└── dev/, proc/, sys/       → Kernel and device interfaces (like Linux)
```

## Task 4 | Android - Forensics Artifacts

Forensics Artifacts Before we even begin any investigation, we need to know what artifacts are important from a forensics point of view and where they are located within the Android device. Once we understand, we can handpick them, unbox them, and extract the information that we find interesting about the case we are investigating. Let's explore some key artifacts in an Android device, their forensic value, and where they are located.

 **SMS/MMS & Call Logs**  **Explanation**

 Communications are important in any forensics case, and SMS / Call logs can be crucial evidence. On an Android device, they can be found in a database that stores the content and metadata of messages sent and received through the device's messaging application.

 **Location**

 
- /data/data/com.android.providers.telephony/databases/mmssms.db (for SMS/MMS)

 ![Image 9](https://tryhackme-images.s3.amazonaws.com/user-uploads/5e8dd9a4a45e18443162feab/room-content/5e8dd9a4a45e18443162feab-1747026272461.png)

 
- /data/data/com.android.providers.contacts/databases/calllog.db (for Call Logs)

 ![Image 10](https://tryhackme-images.s3.amazonaws.com/user-uploads/5e8dd9a4a45e18443162feab/room-content/5e8dd9a4a45e18443162feab-1747026477450.png)

 **Forensics Value**

 
- Recovers complete text message content
- Provides sender/receiver numbers and timestamps
- Supports conversation reconstruction and timeline validation
- Useful for mapping user communication patterns

 **Contacts & Address Book**  **Explanation**

 Contact details on an Android device can also be crucial evidence in any forensic case. This database contains all contacts saved on the device or synced through accounts.

 **Location**

 
- /data/data/com.android.providers.contacts/databases/contacts2.db

 ![Image 11](https://tryhackme-images.s3.amazonaws.com/user-uploads/5e8dd9a4a45e18443162feab/room-content/5e8dd9a4a45e18443162feab-1747026523849.png)

 **Forensics Value**

 
- Identifies social connections and associates
- Provides potential leads through phone numbers and emails
- Supports triangulation with call logs and messages

 **Browser History**  **Explanation**

 These databases record detailed web activity, including visited URLs, page titles, visit timestamps, and search terms the user enters. Depending on the browser, they may store cookies and autofill form data. This information helps identify the user's online interests and behavioral patterns.

 **Location**

 
- **Chrome:**  
- /data/data/com.android.chrome/app_chrome/Default/History

 ![Image 12](https://tryhackme-images.s3.amazonaws.com/user-uploads/5e8dd9a4a45e18443162feab/room-content/5e8dd9a4a45e18443162feab-1747026627018.png)

 **Forensics Value**

 
- Reveals internet usage habits and search interests
- Can indicate planning or intent (e.g., searches about crimes or escape routes)
- May expose accessed web services like cloud storage or forums

 **Location Data**  **Explanation**

 Devices often record detailed GPS and network-based location data through Google Play Services. These databases track location visits, movements, place labels, and timestamps, usually even when apps are not actively used.

 **Location**

 
- /data/data/com.google.android.gms/databases/ (multiple files)
- location.db, networklocations.db, or com.google.android.location

 **Forensics Value**

 
- Maps user movements and frequented places
- Helps confirm or refute the presence at a crime scene
- Crucial in reconstructing past activities and routes

 **Photos, Videos, & Metadata**  **Explanation**

 Photos and videos taken with the device often store metadata like timestamps, device model, and geolocation in EXIF headers. Additionally, media folder structures can indicate third-party sharing apps or camera sources.

 **Location**

 
- /sdcard/DCIM/
- /sdcard/Pictures/
- /sdcard/WhatsApp/Media/

 ![Image 13](https://tryhackme-images.s3.amazonaws.com/user-uploads/5e8dd9a4a45e18443162feab/room-content/5e8dd9a4a45e18443162feab-1747026695824.png)

 **Forensics Value**

 
- Provides visual timelines
- Can include GPS stamps and timestamps
- Sometimes, it shows deleted images in thumbnails or previews

 **Instant Messaging Apps**  **Explanation**

 The third-party messaging apps store messages, media, call logs, and user metadata in private app directories, usually in SQLite databases or encrypted storage. The artifacts may include chat logs, group memberships, timestamps, and shared file references, even after deletion.

 **Location**

 
- /data/data/com.whatsapp/databases/msgstore.db
- /sdcard/WhatsApp/Media/ (for images, voice notes, etc.)

 **Forensics Value**

 
- Direct evidence of conversations
- Highly relevant in criminal or civil investigations
- Media attachments often serve as proof in any forensic case
- Exposes communication across encrypted and non-encrypted channels

 **Application Data**  **Explanation**

 Beyond preferences and databases, many apps create custom logs, usage traces, and artifact-rich directories containing JSON, XML, and plain-text files. These files often include crash logs, analytics pings, locally cached content, and background job histories.

 **Location**

 
- /data/data/[app.package.name]/
- For example: /data/data/com.instagram.android/ or /data/data/com.snapchat.android/

 ![Image 14](https://tryhackme-images.s3.amazonaws.com/user-uploads/5e8dd9a4a45e18443162feab/room-content/5e8dd9a4a45e18443162feab-1747026749228.png)

 **Forensics Value**

 
- Can uncover app usage patterns
- Exposes background activity not visible in UI
- Hidden data like cached credentials, logs, or usage stats
- Examining a suspicious app

 **User Accounts & Google Services**  **Explanation**

 Android stores information about accounts synced with the device, including Google, social media, email, and app-specific logins. Account data includes login timestamps, service tokens, sync settings, and sometimes unhashed usernames or emails.

 **Location**

 
- /data/system/users/0/accounts.db
- /data/data/com.google.android.gms/databases

 ![Image 15](https://tryhackme-images.s3.amazonaws.com/user-uploads/5e8dd9a4a45e18443162feab/room-content/5e8dd9a4a45e18443162feab-1747026839996.png)

 **Forensics Value**

 
- Identifies user identities and linked online services
- Maps services used (e.g., Gmail, Dropbox, and Instagram)
- Reveals syncing frequency and potential data exfiltration vectors
- Supports correlation with web or cloud-based investigations

 **Installed Applications Information**  **Explanation**

 This artifact contains metadata about all applications installed on the device, including app names, package IDs, version numbers, install times, and permission settings. It includes system and user-installed apps, offering insight into the device's overall software environment.

 **Location**

 
- /data/system/packages.xml

 ![Image 16](https://tryhackme-images.s3.amazonaws.com/user-uploads/5e8dd9a4a45e18443162feab/room-content/5e8dd9a4a45e18443162feab-1747026904324.png)

 **Forensics Value**

 
- App usage over time
- Can support a timeline or behavioral pattern
- Evidence of app installs/uninstalls
- Lists dangerous permissions that could relate to surveillance or exploitation.

## Task 5 | Tools for the Trade

We all know that digital devices like smartphones know and contain more information about us than anything else. From our location and messages to photos, browser history, and even how long we stare at an application, all the information is stored in these devices. Whether a device is found in a criminal case or confiscated from a disgruntled employee suspected of stealing company secrets.

 So when investigators or digital forensic analysts try to investigate an Android device, whether for a legal case, breach investigation, or insider threat, they need the right tools to extract information from the device without compromising the evidence or its integrity.

 Getting the Data Off the Phone (Acquisition Tools) Before we analyze the device, we need access to the data. Android devices have layers of security like screen locks, encryption, and app sandboxing. Depending on the situation, we might go for different levels of acquisition:

 
- **Logical:**  Like using a backup to get user-level data.
- **File system:**  Gives access to folders and system files. It allows deeper access than logical acquisition, including app directories and internal storage, which is useful for examining app data and configurations.
- **Physical:**  A bit-by-bit copy of the device's entire memory, including deleted data, unallocated space, and system areas inaccessible through higher-level acquisitions.

 Tools for the Trade The following is a list of tools that can be used in different phases of Android analysis, whether for data acquisition, memory acquisition, or analysis.

    **Tool Name**  **Type / Function**  **Use Case**      **ALEAPP**  Artifact Parser Extracts and parses key Android artifacts (e.g., app data, location, usage stats).   **Autopsy + Android Modules**  GUI Forensics Suite Examines logical dumps with plugin support for call logs, messages, and app data.   **Cellebrite UFED**  Commercial Mobile Forensics Platform Physical, file system, and logical Android (and iOS) device extraction.   **Magnet AXIOM**  Commercial Suite Ingests Android images and categorizes artifacts using timeline, chats, media, etc.   **Oxygen Forensic Detective**  Commercial All-in-One Tool Recovers deleted data, decrypts apps, and parses communications and cloud data.   **ADB (Android Debug Bridge)**  Command-line Interface Communicates with the Android device, collects logs, and performs manual file extraction (root may be needed).   **TWRP Recovery**  Custom Recovery Interface Used to boot into a custom recovery mode for image acquisition or data access.   **LiME (Linux Memory Extractor)**  Memory Dumping Tool It captures volatile memory (RAM) from an Android device and is used for live memory forensics.   **Andriller**  Device Analysis + Unlock Tool Extracts device data, PIN cracking, and report generation from Android backups.   **ADB-Backup Extractors**  Backup Utilities Converts Android ADB backups (`.ab` files) into accessible `.tar` archives.   **Protobuf Parsers**  Data Format Parser Parses Android usage stats, app events, and settings stored in Protobuf format. ## Task 8 | Framing Questions

When examining an Android device for forensic purposes, framing the right questions is crucial to guiding the investigation, prioritizing evidence acquisition, and preserving the integrity of digital data.

 General Questions Every forensic examination can benefit from questions framed using the **5Ws and H**  method:

 
- **Who**  used the device?
- **What**  activity took place?
- **When**  did it occur?
- **Where**  was the device used?
- **Why**  was the activity performed?
- **How**  was the activity executed (technically)?

 Scenario 1: Insider Data Theft Let's assume a disgruntled employee was not promoted recently because of poor performance. He has been seen making attempts to access sensitive files, and it is doubtful that he may be sending confidential company documents to a competitor using communication apps.

 Keeping this scenario in mind, we will frame the following forensic questions to be asked/looked for on his confiscated device.

 **Forensic Questions to Frame**

 
- Who had access to the WhatsApp account on the device?
- What documents or files were shared through the app?
- When were the messages sent and received?
- Which contacts were involved in the communication?
- Was WhatsApp used exclusively, or were other communication apps like Telegram or Signal involved?
- Were screenshots or screen recordings taken of sensitive information?
- Is there any indication of deleted messages or use of "disappearing messages"?

 **Where to Find These Questions?**

 To find the answers to these questions, we will extract and examine the following evidence:

 
- WhatsApp db
- Other communication apps
- Evidence of deleted messages
- Chat logs
- Evidence of presence / deleted communication apps, etc

 Scenario 2: Visiting Malicious Websites or Engaging in Cybercrime An individual is suspected of accessing dark web sites for illegal transactions.

 **Forensic Questions to Frame**

 
- Which browsers or apps were used to access the web (Chrome, Firefox, Tor)?
- What URLs were accessed, and what was their content?
- Were VPNs or proxy apps used to mask location?
- Are there downloaded files or cached web content stored on the device?
- When was the suspicious content accessed?
- Is there evidence of cryptocurrency apps or wallets?

 **Where to Find These Questions?**

 To find the answers to these questions, we will extract and examine the following evidence:

 
- Browser history and web activity
- Web cache and downloads
- Use of anonymous or privacy-focused browsers
- Communication apps linked to web-based crime

 Scenario 3: Malware Infection or Device Compromise An Android phone is suspected of being infected with spyware.

 **Forensic Questions to Frame**

 
- What apps are installed? Are there any from unknown sources?
- When did the suspicious activity (e.g., overheating, battery drain) occur?
- Is there evidence of remote access or backdoors?
- What system logs or permissions changes were recorded?
- Did the device connect to unfamiliar IPs or perform data exfiltration?

 **Where to Find These Questions?**

 To find the answers to these questions, we will extract and examine the following evidence:

 
- Installed app list and APKs
- App permissions and behavior
- Unknown or background-running services
- Malware traces in logs and crash reports
- Communication with suspicious apps

 In a nutshell, when investigating a forensic case involving an Android device, we need to understand what artifacts can leave the footprints of the activities related to that scenario. We identify those artifacts and examine them to find answers to the framed questions that could help solve the case. Another important thing to note is that framing the questions is important in any forensic investigation. It provides a clear direction as to what we are looking for.
