## Task 2 | Log Insights: Scheduled Tasks and Services

![Image 2](https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/65689dd0aa8385543a939d8bdde6dc90.png)

 Scheduled tasks on Windows are automated events designed to execute actions at predetermined times or intervals. While primarily used for system automation or routine tasks, threat actors often leverage this feature to establish persistence by implanting malicious commands and executing them regularly.

Events Generated by Scheduled Tasks

When configured on a machine, Windows Event Logs provide enough visibility to monitor and discover unusual creations and modifications of scheduled tasks. Below are the notable events generated by scheduled tasks when being leveraged maliciously.

- Event ID `4698` - A scheduled task was created.
- Event ID `4702` - A scheduled task was updated.

These events can be viewed using Windows Event Viewer (`eventvwr.msc`) or command-line tools like `wevutil.exe` and PowerShell's `Get-WinEvent / Get-EventLog`.

From the events generated by these specific event IDs, the following information can be collected:

**Artefact** **Example** Event timestamp
`Feb 09, 2024 09:35:03 UTC`

The user who created the scheduled task
`wkstn\victimuser`

The commands to be executed
`C:\Windows\Temp\update.exe`

Execution interval
`Daily every 9:00 AM`

User context of the execution
`Execute as NT Authority\SYSTEM`

**NOTE: The values above are only examples and do not reflect the malicious events created in the interactive VM.**

For a quick example, let's inspect the events generated by scheduled tasks through **Windows Event Viewer** . Press `CTRL+R` (or access the Run dialogue box from the taskbar) and type `eventvwr.msc` to open the application. Then, navigate to `Windows Logs > Security` and filter by using the event IDs above.

![Image 3](https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/78691862af7a1ecbf357c8df76ed59ba.png)

Once the filter is applied, investigate each event log and assess for unusual creation or modification of scheduled tasks. For some pointers, here are some ways to determine if the scheduled task is unusual.

1. **Command to be executed**  - If noticeable to the naked eye, the commands executed by the scheduled task can already answer whether it is malicious activity. Otherwise, examine the exact binary to be executed to determine the correct verdict.
2. **User Context**  - Based on the user context, whether the user account is expected to create or modify a scheduled task can be determined. In a typical environment, only system administrator accounts are known to deploy or change them.
3. **Execution context**  - Similar to item #2, assessing the context in which the scheduled task runs is essential, like the user account under which it executes and its required privileges. Tasks running under privileged accounts may be indicative of malicious activity.
4. **Trigger interval**  - Using the interval of task execution, outliers can also be determined by analysing how frequently or when the task will execute. Some threat actors use typical triggers, such as execution every 3 hours, when a specific user logs on, or during a particular time.
5. **Comparison to baseline**  - If available, compare the deployment of new scheduled tasks in other machines to see if it is normal. When the activity is an actual deployment by a system administrator, multiple machines should also have the same event. However, this still depends on the commands to be executed by the task since this behaviour can also be an activity done by a threat actor.

 ![Image 4](https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/8623cd551ceb31567d4d30db86ab9c5c.png)

Events Generated by Services

Moving on, Windows services are background processes responsible for various system functions. Despite their essential role in the system's operations, it is also often used by threat actors in establishing persistence similar to scheduled tasks. This is due to their functionality being executed during the machine's startup.

Again, the notable events generated when services are being abused can also be seen through Windows Event Logs from the following events:

- Event ID `7045` - New Service was installed (System Channel).
- Event ID `4697` - A service was installed in the system (Security Channel).

From the events generated by these specific event IDs, the following information can be collected:

**Artefact** 
**Example** 
Event timestamp
`Feb 09, 2024 09:35:03 UTC`

The user who created the service
`wkstn\victimuser2`

The service binary to be executed
`C:\Windows\System32\malicious_service.exe`

Service start type
`Automatic`

User context of the execution
`Execute as NT Authority\LocalService`

NOTE: The values above are only examples and do not reflect the malicious events created in the interactive VM.

Similarly, let's inspect the events generated in the Windows Event Logs. However, in this instance, we will use PowerShell commands to view the logs related to Windows services. Given that, fire up a PowerShell instance and use the command below to list service creation events.

    Administrator: Windows PowerShell  
```bash
# System Channel = Event ID 7045
PS C:\Users\Administrator> Get-WinEvent -FilterHashTable @{LogName='System';ID='7045'} | fl

# Security Channel = Event ID 4697
PS C:\Users\Administrator> Get-WinEvent -FilterHashTable @{LogName='Security';ID='4697'} | fl
```

  Once executed, you will see the output rendered in a list view. Using the output, you can start analysing the services installed individually. Like the scheduled tasks section, we will also discuss ways to determine if the service is unusual.

1. Command to be executed - Unlike scheduled tasks, Windows services use a specific binary tailored to run as a service. Given this, assessing whether the binary attached to a service is malicious is necessary. Moreover, there are some cases wherein threat actors use a one-liner command to execute a malicious command during the machine's startup, which gives an obvious indicator to the naked eye since services are only expected to run service binaries.
2. User Context - The user context determines whether the account is expected to create or modify a service. In a typical environment, only system administrator accounts are known to deploy or modify them.
3. Comparison to baseline - If available, compare the deployment of new service in other machines to see if it is normal. When the activity is an actual deployment by a system administrator, multiple machines should also have the same event. However, this still depends on the binary to be executed by the service since this behaviour can also be an activity done by a threat actor.

- Based on the **Location** , combine the observations above and see if there are potential anomalies. Typically, the location set will be `\`**** if the location is not specified in the scheduled task's name during its creation.

Some threat actors leverage these settings to deploy the malicious scheduled tasks more persistently and covertly, and some use cases are:

- **Stop after a specific time**  - This allows the task to stop its malicious activities and hide itself after its successful execution.
- **Auto deletion**  - This allows the malicious scheduled task to automatically clean itself up to remove any traces within the compromised machine.

Utilising Command Line Utilities

In the previous sections, we have used the Task Scheduler application and Notepad to inspect and analyse the scheduled tasks. Now, let's divert into command-line utilities to optimise the investigation methodology.

 For starters, fire up a PowerShell window and execute the command below to list all enabled scheduled tasks.

   Administrator: Windows PowerShell  
```bash
PS C:\Users\Administrator> Get-ScheduledTask | Where-Object {$_.State —ne "Disabled"}
```

  Alternatively, you can also use **schtasks.exe**  to get the same information. The command below lists all scheduled tasks, renders them in a CSV format and excludes all disabled tasks using **findstr.exe** .

    Administrator: Windows PowerShell  
```bash
PS C:\Users\Administrator> schtasks.exe /query /fo CSV | findstr /V Disabled
```

  Next, let's list all enabled scheduled tasks sorted by their creation date. Moreover, we will output the creation date and author to add more context to each task.

    Administrator: Windows PowerShell  
```bash
# List of all scheduled tasks, sorted by their creation date
PS C:\Users\Administrator> Get-ScheduledTask | Where-Object {$_.Date —ne $null —and $_.State —ne "Disabled"} | Sort-Object Date | select Date,TaskName,Author,State,TaskPath | ft
```

  **NOTE: To render the output in list view, replace the ft (Format-Table) command above with fl (Format-List).**

Based on the output of the initial command, you may already observe some notable scheduled tasks. To view the additional details for a specific task, you may use the following commands to view its properties.

**Details** **Command** Command to be executed`(Get-ScheduledTask -TaskName REPLACE_WITH_ACTUAL_TASKNAME).Actions`

Execution Triggers`(Get-ScheduledTask -TaskName REPLACE_WITH_ACTUAL_TASKNAME).Triggers`

User Execution Context`(Get-ScheduledTask -TaskName REPLACE_WITH_ACTUAL_TASKNAME).Principal`

**Note: We have enclosed the Get-ScheduledTask commands in parenthesis since we are accessing its Actions/Triggers/Principal properties.**

Lastly, we can combine all these into a single script to list all scheduled tasks sorted by their creation date, together with all the significant information.

    Administrator: Windows PowerShell  
```bash
# List all enabled scheduled tasks with creation date and command to be executed, sorted by date and printing all additional information
$tasks = Get-ScheduledTask | Where-Object {$_.Date —ne $null —and $_.State —ne "Disabled" —and $_.Actions.Execute —ne $null} | Sort-Object Date

foreach ($task in $tasks) {
    $taskName = $task.TaskName
    $taskDate = $task.Date
    $taskPath = $task.TaskPath
    $taskAuthor = $task.Author
    $taskCommand = $task.Actions.Execute
    $taskArgs = $task.Actions.Arguments
    $taskRunAs = $task.Principal.UserId

    # Output service information
    Write-Host "Task Name: $taskName"
    Write-Host "Task Author: $taskAuthor"
    Write-Host "Creation Date: $taskDate"
    Write-Host "Task Path: $taskPath"
    Write-Host "Command: $taskCommand $taskArgs"
    Write-Host "Run As: $taskRunAs"
    Write-Host ""
}
```

Manual Inspection of Services

Again, Windows Event Log analysis only applies when the machine is configured to log all service creation events. Given this prerequisite, we will proceed with the discussion of analysing services without relying on Windows Event Logs. First, let's leverage the GUI version of Windows Services by using `CTRL+R` (or access the Run dialogue box from the taskbar) and typing `services.msc`.

You might be curious why we are using this approach. The reason is that threat actors tend to implant a persistent service that must be run automatically during startup.

Let's use the Amazon SSM Agent service as an example. Double-click on the service to view the details.

![Image 16](https://tryhackme-images.s3.amazonaws.com/user-uploads/5dbea226085ab6182a2ee0f7/room-content/e67b3c1a1c56e3aa9b4ee79c57f87597.png)

- The service executable is named **amazon-ssm-agent.exe** .
- The startup type is set to `Automatic`.
- The user context during execution is set to `LocalSystem`.

Lastly, the Recovery tab also provides additional insights regarding the behaviour of the service when it fails. Given the example of Amazon SSM Agent, this setting may not look suspicious, but this configuration can be leveraged maliciously. Some potential use cases would be:

- Run a program during the first failure to covertly execute a malicious command different from the visible one in the General tab.
- Restart the service on subsequent failures to ensure that the persistence mechanism will be re-executed if it fails.
- Restart the service after a few minutes to ensure the malicious implant is alive.

Utilising Command Line Utilities

In the previous sections, we have used multiple GUI applications to inspect and analyse Windows services. Now, let's divert into command-line utilities to optimise the investigation methodology.

For starters, fire up a PowerShell window and execute the command below to list all running services set to run automatically.

    Administrator: Windows PowerShell  
```bash
PS C:\Users\Administrator> Get-Service | Where-Object {$_.Status -eq "Running" -and $_.StartType -eq "Automatic"}
```

  Next, let's add the binary these services execute using the command below.

    Administrator: Windows PowerShell  
```bash
PS C:\Users\Administrator>
$services = Get-Service | Where-Object {$_.Status -eq "Running" -and $_.StartType -eq "Automatic"}

foreach ($service in $services) {
    $serviceName = $service.Name
    $serviceDisplayName = $service.DisplayName
    $serviceStatus = $service.Status
    $serviceWMI = (Get-WmiObject Win32_Service | Where-Object { $_.Name -eq $serviceName })

    Write-Host "Executable Path: $servicePath"
```

  You may have observed that most services run the `svchost.exe` binary, which is common for Windows services. Moreover, the location of the service binaries is set to known program locations, which is also a good way to assess whether the service is malicious from a quick glance. There are typical cases where the service binaries are stored in an unusual directory, such as AppData or Temp, where attackers initially store their payloads. With this approach, assessing the service based on directory anomalies of service binaries is noteworthy.

Next, there might be cases where the malicious service has stopped after running its initial payload, like when the service binary immediately migrates into another process by injecting a malicious shellcode. Given this, reviewing all the stopped services that run automatically on boot is also essential to determine potential outliers.

    Administrator: Windows PowerShell  
```bash
PS C:\Users\Administrator> Import-Module C:\Tools\Get-RegWriteTime.ps1;
$services = Get-Service | Where-Object {$_.Status -eq "Stopped" -and $_.StartType -eq "Automatic"}

foreach ($service in $services) {
    $serviceName = $service.Name
    $serviceWMI = (Get-WmiObject Win32_Service | Where-Object { $_.Name -eq $serviceName })
    $serviceDisplayName = $service.DisplayName
    $serviceStatus = $service.Status
    $servicePath = $serviceWMI.PathName
    $serviceUser = $serviceWMI.StartName

    Write-Host "Display Name: $serviceDisplayName"
    Write-Host "Service Status: $serviceStatus"

    Write-Host "User Context: $serviceUser"
    Write-Host "Last Write Time: $serviceLastWriteTime"
    Write-Host ""
}
```

You may have observed that we have focused the analysis on services set to run automatically on startup. This approach speeds up the analysis of services expected to run automatically on boot. However, the caveat here is that there might be cases wherein the malicious services are covertly hidden by setting it in a disabled or manually triggered state and only executed by another implant.

Given this, it is still important to review and assess all the services by relying on the methodology discussed in the previous task to determine potentially malicious ones or visible outliers compared to a known good baseline.

Browser Forensics 101

As mentioned, browser forensics is crucial in digital forensics and incident response. It provides valuable insights into a user's online activity, including websites visited, login credentials, download history, bookmarks and extensions or plugins.

First, let's run down a few common artefacts and elaborate on some use cases that would give us quick wins in determining notable actions worthy of investigation.

**Artefact** **Scenarios** **Browsing History** Since this artefact provides the exact site's URL and the time and date of the visit, the user's online activities can be easily plotted, whether related to access to a malicious link or unauthorised internal documents.
Downloads

Sessions
Leveraging this artefact allows you to combine all the user's activity in a specific timeframe or a single browsing session. By knowing this, it is easy to correlate the sites accessed by the user and determine the pattern of the activity.
Extensions
Certain threat actors now leverage fake browser extensions and await for users to deploy and compromise users. Examples such as extensions harvesting login data or recording screen activity are prominent nowadays.

Based on the contents of the output file, we can classify the information using the value of the record_type field. For each type, we will focus on the significant fields that contain valuable information for our investigation:

**Record Type** **Field Values** contact
- `displayName` - Provides the given name of the email account, which is typically the first and last name of the user.
- `email` - Email address of the contact.
- `mri` - Unique identifier of the contact.
- `userPrincipalName` - Contains the user principal name of the contact, which is typically the email of the associated user.

message
- `conversationId` - Unique identifier of the conversation thread. Combining the messages with the same ID can retrieve the complete message thread.
- `composetime / createdTime` - Timestamp when the message is created.
- `content` - The actual message sent.
- `creator` - The message's sender in the MRI value. This can be translated by correlating the value in the contact record_type.
- `isFromMe` - Provides the direction of the message, whether it was sent or received by the owner of the Teams metadata.
- `properties` - Contains additional parameters, such as time of edit (when the message is edited) or file attachment details.

You may have observed that it is tedious to inspect the contents of the output manually. But don't worry; from here, we will use some PowerShell commands to start the analysis. First, let's parse all the contact information and store the MRI and userPrincipalName in a hashtable. This will be used as a lookup table to correlate the creator's value in each message. Once you execute the command below, you will see the list of MRI and userPrincipalName key-value pairs.

    Administrator: Windows PowerShell  
```bash

$users = @{}

# Initialise user hashtable for correlation
foreach ($data in $teams_metadata) {
   if ($data.record_type -eq "contact") {
     $users.add($data.mri, $data.userPrincipalName)
   }
}
$users | fl
```

  Then, we want to combine the messages with identical conversation IDs to view the messages in each conversation thread.

    Administrator: Windows PowerShell  
```bash
PS C:\Users\Administrator> $teams_metadata = cat .\output.json | ConvertFrom-Json
$messages = @{}
# Combine all conversations/messages with the same ID
foreach ($data in $teams_metadata) {
  if ($data.record_type -eq "message") {
    if ($messages.keys -notcontains $data.conversationId) {
      $messages[$data.conversationId] = [System.Collections.ArrayList]@()
    }
    $messages[$data.conversationId].add($data) > $null
```

## Task 10 | Deep-diving on Office Applications III: Microsoft OneDrive

Microsoft OneDrive Analysis

The last application to be analysed for this task is OneDrive, a known cloud storage service provided by Microsoft. Given the nature of the application, we will focus on analysing files synced locally and determine any unusual indicators. First, the OneDrive artefacts are stored in the user's `Appdata\Local\Microsoft\OneDrive\logs` directory. This directory contains additional directories, such as Business* and Personal, which differ only by the subscription of the account used. Microsoft products are licensed in a typical enterprise environment, so the artefacts are commonly seen in the Business folder.

From the Business folder, two important files contain significant information:

- `SyncEngine.odl` - Contains all operations performed, including the information of each file processed. The `odlgz` files are the compressed files of SyncEngine.odl.
- `SyncDiagnostics.log` - A log file that contains the diagnostic information related to the synchronisation process of OneDrive. It stores details about sync activities, such as download and upload progress, bytes uploaded or downloaded, and the number of files processed.

You may have observed that the `SyncDiagnostics.log` is only a text file that can be easily analysed. However, the `SyncEngine.odl` files require additional parsing to be analysed.

To parse the OneDrive artefacts inside the ODL files, we will use [OneDriveExplorer](https://github.com/Beercow/OneDriveExplorer), another open-source tool hosted in GitHub. Note that the tool must be run as administrator since it will try to scan the OneDrive directories of each existing user in the machine.
